Development of The Rabid CF Developer's AJAX Framework also known as Rabid_AJAX™ is proceeding nicely. I have worked out the final details on how-to bootstrap the system using the minimum amount of JavaScript code possible meanwhile maintaining 100% privacy for all JavaScript used by the system. I have also been working on the design of the AJAX interface and I have found-out it would be quite nice if the AJAX system would allow certain parameters to be made persistent between AJAX Sessions.

I have been using Rabid_AJAX™ to create my first real operational application that edits and maintains the contents of several database tables. When it comes time to perform certain sub-functions such as committing an edited record back to the database it would be useful to not have to reissue the list of arguments that were previously issued to establish the operational context for the AJAX Sessions that are part of the same area of functionality. For instance, several functions such as adding a record as well as saving and deleting records all share one thing in common - the callBack JavaScript function. The common callBack function for all these functions does the same thing - namely to simply refresh the list of records along with the GUI elements that are used to edit the data. It would make for an easier to code application if I didn't have to jump through huge hoops just to get the parameters passed around to all the various AJAX Session invocation points although the same goal could be accomplish by using a global data structure of some kind to store the commonly used parameters I would rather build this into the AJAX API so that a context could be established and used until a different context is needed.

I have also decided to build into Rabid_AJAX™ an Object Broker. An Object Broker would be responsible for maintaning a list of JavaScript Objects that have been previously downloaded from the server. Since there is no need to continuously download the same Objects over and over when moving from one functional area to another it would be more optimal to create an Object Broker. The Object Broker will communicate the list of previously downloaded Objects to the server wheneve an AJAX function is initiated. The server will then figure-out which objects to download to the browser based on those functions that were previously downloaded and those Objects that must be present in the client to support the functionality being downloaded to the client. The Object Broker will ensure the same Objects are downloaded only once unless the Object has changed since the last time it was downloaded from the server. Each Object will have a serial number that defines the date and time the Object was checked-into the system. Each time an Object is updated the serial number is adjusted. Whenever an Object's serial number in the Server differes from that found in the client the updated Object will be downloaded as-needed to the client during the next AJAX Session.

Because Rabid_AJAX™ is meant to be the foundation for a Content Management System it may be advantageous to manage HTML Content and Data using the same technique as an Object Broker but for HTML content or XML content or raw data which is also a form of content.

Rabid_AJAX™ will allow web based applications to become completely dynamic for both JavaScript (which is typically considered static - downloaded once for each page invocation but non-changing while the page is sitting in the browser no matter how long the page is sitting in the browser) and HTML content. The ablity to turn JavaScript into a dynamic form of content will truly make Rabid_AJAX™ into a unique product, at least this is the hope I have as the designer and architect.
