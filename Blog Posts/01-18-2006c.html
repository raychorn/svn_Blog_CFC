The real reason ODBC DSN's don't like <cfqueryparam> is because ODBC drivers don't support <cfqueryparam> so CF simply chokes and barfs when ODBC meets <cfqueryparam>. Okay, that's cool - I'll just restrict all the code I write that uses <cfqueryparam> to native SQL Server DSN's. Groovy.

Well this is why I do NOT use <cfqueryparam> when I am coding for clients who pay me money to craft CF apps for them. Simply put, I like to remain employed over the long haul and I know if I try to explain to those who sign my weekly timesheets that the CF apps they just paid me to code only work with native SQL Server drivers I'm gonna be shoved out the nearest airlock in nothing flat. So I ride the rapids and I do not use <cfqueryparam> I also don't allow URL scoped variables to be injected into the SQL code I craft.

So I coded my own CF UDF's to perform the same function as <cfqueryparam> so my code will work with any DSN's my clients wish to use - as it should be.

So now you can make-up your own mind on this issue. Some may try to tell you that you must use <cfqueryparam> in order to keep your database safe from SQL injection attacks however if you do use <cfqueryparam> then your code ain't gonna work for just any DSN's you may wish to use.
