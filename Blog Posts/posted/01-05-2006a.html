To Default of not to Default - This is the question "some" developers want answered.

Okay, so I just learned something "new" about ColdFusion that I had always known but never used and hwre's why...

Apparently the ColdFusion Administrator allows Client Variables to be defined in one place as a "default" that all Applications can then use. No problem with doing that, until... Well, until you end up working for some companies, such as I do from time to time, who just don't want to let lowly developers get access to the ColdFusion Administrator for whatever odd reasons. When you cannot see "how" the ColdFusion Administrator is setup and therefore cannot see "where" your Client Vars are going to live it may not be all that wise to simply let the default be the preferred method for handling Client Vars.

Okay, so what's wrong with using the "defaults" again ?

Defaults are fine, in my mind, right up until the point in time comes for the code we each write as individuals to be given to others to work with. It is my firm feeling that the code I write should be self-documenting. Self-documenting code should be its very nature be coded in such a way so as to remove the need for anyone to know what the defaults are that the code depends upon. My own personal preference for Client Vars is to use the database to store them. So when I code my ColdFusion applications is always code a <cfapplication> tag that states specifically how my code likes to be run including the DSN for the Client Vars. Oh sure, I could set this up once in the ColdFusion Administrator and then never touch it again, ever, but will anyone else on the planet I might wish to give me code to know how or why this was done ? I doubt it. Also as a software engineer I don't like to trust "defaults" because they can change and I trust what I have control over when it comes to code I work on. And I have worked on software coded by others that used certain undocumented "defaults" that caused the code to fail simply because nobody knew what those default values were supposed to be and since I did not know what they were supposed to be I might stupidly make a minor change tha caused the code to cease to function with no idea of what caused the failure(s). Been there and done that.

So then I really didn't remember that the Client Vars DSN "could" be setup in the Administrator and I really didn't care either since I always do setup a DSN using the Admnistrator as the "default" which ends up being a safety net for me in case I inherit someone else's code who used the "default" but I also specifically state the Client Vars DSN in all my <cfapplication> tags to capture the fact that I am using Client Vars and oh by the way here's the DSN(s) I was using.
